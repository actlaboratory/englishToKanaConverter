# 英語→カナ変換ロジック解説

## 1. 目的・対象読者

本ドキュメントは、`englishToKanaConverter/englishToKanaConverter.py`に実装されている英語文字列→カタカナ変換アルゴリズムの内部動作を解説するものである。

辞書データ自体の形式（キー・値の制約、ファイル構成）についてはREADME.mdの「辞書のメンテナンス」節、およびJSON化の経緯については[issue15-dictionary-json-format.md](./issue15-dictionary-json-format.md)を参照。本ドキュメントは**アルゴリズム（どの辞書を、どういう順序で、どういう規則で使うか）**に焦点を当てる。

対象読者は、変換結果が期待通りにならないケースを調査する開発者、および辞書のメンテナンス方針（どのファイルに何を追加すべきか）を判断する必要がある開発者を想定する。

## 2. 全体像

`EnglishToKanaConverter.process(s, spellout=True)`（`englishToKanaConverter.py:288`）が公開エントリーポイントであり、内部的に以下の順序で処理を行う。

```
process(s)
 ├─ 1. _zenToHan(s)                 全角アルファベットを半角に変換
 ├─ 2. _splitUpperCase(s)           文字列を「単語らしきまとまり」のリストに分割
 ├─ 3. リストの各要素について:
 │     ├─ _engToKana(chunk)         辞書ベースの変換（WORDS / PHRASES / PREFIX / SUFFIX）
 │     └─ _romanToKana(chunk)       上記で変換できなかった部分をローマ字読みで変換
 ├─ 4. _trimWhitespaceBetweenUpperCase(list)  リストを結合して1つの文字列に戻す
 └─ 5. _alphaToSpell(s)             それでも変換できなかったアルファベットを1文字ずつスペルアウト
                                     （spellout=Falseの場合はスキップされる）
```

ポイントは、**各段階が「前段階で変換しきれなかった残り」だけを処理する保険の連鎖になっている**という点である。辞書に単語として登録されていれば1の時点で高品質な変換ができ、登録されていなくてもローマ字として読める形であれば3のローマ字変換で対応でき、それでもダメなら最終手段として4でアルファベット読み（スペルアウト）される。README.mdに書かれている「変換後の文字列にはアルファベットが一切出てこなくなる」という保証は、この最終段階のスペルアウトによって担保されている。

## 3. 前処理

### 3.1 全角→半角変換（`_zenToHan`, L27-31）

`ZENHAN_TABLE`（`constants.py`）を使い、全角アルファベットと全角アポストロフィーを半角に変換する。以降の処理はすべて半角アルファベットを前提とする。

### 3.2 大文字列の分割（`_splitUpperCase`, L33-90）

英大文字が連続する箇所を検出し、文字列を複数のチャンク（リストの要素）に分割する。この分割は、後続の辞書引き処理の単位を決めるための重要な前処理であり、**キャメルケースの単語境界の推定**と**略語（頭字語）の検出**を同時に行っている。

大文字の連続（`[A-Z]+`の1つのまとまり）ごとに、長さに応じて次のいずれかとして扱われる。

| ケース | 条件 | 挙動 |
| --- | --- | --- |
| ① 通常の単語境界 | 連続長が1文字 | 分割しない。その大文字位置から末尾（または次の境界）までをまとめて1チャンクとする |
| ② 頭字語として展開 | 連続長が2〜3文字（`UPPER_MAX=3`以下）かつ`UPPER_IGNORE`に**含まれない** | 1文字ずつ独立したチャンクに分解する（＝最終的に1文字ずつスペルアウトされる） |
| ③ 頭字語として展開しない | 連続長が2〜3文字かつ`UPPER_IGNORE`に**含まれる** | 分割しない。そのまとまりを1チャンクとして辞書引きに回す |
| ④ 長い大文字列 | 連続長が4文字以上（`UPPER_MAX`超） | 直後が小文字なら、最後の1文字を新しい単語の先頭とみなして分離し、残った大文字列を②③のルールで再判定する。直後が小文字でなければ、大文字列全体を1チャンクとする |

`UPPER_MAX`は`constants.py`で`3`に固定。`UPPER_IGNORE`は`constants/upper_ignore.json`（`AND`, `FOR`, `WEB`, `ZIP`など）に定義されており、「3文字以下の大文字列だが、頭字語ではなく普通の単語として読ませたい」例外を登録する場所である。

動作確認済みの具体例（`converter.process()`の実測結果）:

| 入力 | 分割結果（`_splitUpperCase`の出力） | 該当ケース | 最終変換結果 |
| --- | --- | --- | --- |
| `Claude` | `['Claude']` | ① | クロードゥ |
| `iPhone` | `['i', 'Phone']` | ①（`P`の位置で分割） | アイフォウン |
| `NHK` | `['N', 'H', 'K']` | ② | エヌエイチケー |
| `USB` | `['U', 'S', 'B']` | ② | ユーエスビー |
| `AND` | `['AND']` | ③（`UPPER_IGNORE`に登録済み） | アンドゥ |
| `GOOGLE` | `['GOOGLE']` | ④（直後に小文字なし＝分割不要） | グーグル |
| `COVID-19` | `['COVID-19']` | ④（直後が`-`で小文字ではない） | コビッドゥ-19 |
| `HTMLFile` | `['HTML', 'File']` | ④（`F`の前で新語開始、残り`HTML`は4文字なので③④どちらでもなく1チャンクのまま） | エイチティーエムエルファイル |
| `XMLHttpRequest` | `['X', 'M', 'L', 'Http', 'Request']` | ④→②の連鎖（`Request`と`Http`を切り出した後、残り`XML`は3文字なので②が適用され1文字ずつ展開） | エックスエムエルエイチティーティーピーリクウェストゥ |

`HTMLFile`と`XMLHttpRequest`の違いは、大文字列を1文字削った残り（`HTML`=4文字 vs `XML`=3文字）が`UPPER_MAX`を超えるかどうかで、「まとめて1チャンク（④）」になるか「1文字ずつ展開（②）」になるかが変わる点にある。

なお、この段階で1文字チャンクに分解された文字（`N`, `H`, `K`など）は、後述の辞書引き（4章）では1文字だけでは辞書にヒットしないため必ず変換に失敗し、最終的に5章の`_alphaToSpell`でアルファベット読みされる。つまり②のルールは「頭字語を強制的にスペルアウトさせるためのルート」として機能している。

## 4. カナへの変換（本体処理）

`_splitUpperCase`で分割された各チャンクは、`_engToKana` → `_romanToKana`の順に処理される。

### 4.1 単語全体一致（`_engToKana`, L92-118 / `WORDS`辞書）

チャンク内の英字＋アポストロフィーの並び（正規表現`[a-zA-Z']+`）ごとに、まず**大文字化した文字列全体**が`dictionaries.WORDS`に完全一致するかを調べる（`englishToKanaConverter.py:106`）。一致すればその値を採用し、この文字列に対する処理はそこで完了する。

`WORDS`は「単語全体が完全一致した場合にのみ使う」辞書であり、後述の`PHRASES`と異なり、複合語分解の途中の部品としては**一切利用されない**（`_partsToKana`は`PHRASES`／`PREFIX`しか参照しない）。人名など、他の単語の一部分として誤って再利用されると困る語をここに登録する、という使い分けになっている。

### 4.2 複合語・接頭辞・接尾辞分解（`_partsToKana`, L120-183）

`WORDS`に完全一致しなかった場合、`_partsToKana`が呼ばれる。これは**文字列の先頭からできるだけ長く辞書に一致する部分を貪欲に探し、一致した部分の続きを再帰的に処理する**アルゴリズムである。

処理の流れ（対象文字列`s`について）:

1. `cnt`を`len(s)`から`1`まで1ずつ減らしながら、`target = s[0:cnt]`という接頭部分文字列を辞書と照合する。
   - `target`の大文字形が`MUST_SPELLED`（`constants/must_spelled.json`）に含まれる場合、辞書の値を無視して強制的に`_alphaToSpell`でスペルアウトする（L132-135）。**この判定は`PHRASES`の参照より前に行われるため、辞書に読みが登録されていても優先して上書きされる**。
   - `target`が単独の`'`（アポストロフィー）の場合は、それ自体を変換済み（`'`のまま）として扱う。
   - それ以外は`dictionaries.PHRASES`を大文字形で検索する。見つからず、かつ`target`が単語の先頭部分（続きの文字列がまだ残っている）であれば、`dictionaries.PREFIX`（`NO`→ノー、`UN`→アン の2件のみ）も検索する。**`PREFIX`は単語の途中や末尾には適用されない**（`includePrefix`引数で制御、L144）。
2. どの長さの`target`でも一致しなければ、その文字列は変換失敗として扱われ、**元のアルファベット文字列がそのまま**返る（L178-183）。この場合、後段の`_romanToKana`にそのまま渡される。
3. `target`が一致した場合、残り（`remaining = s[cnt:]`）が空ならそこで完了。空でなければ、
   - まず`remaining`全体が`dictionaries.SUFFIX`（`'S`, `D`, `ED`, `ES`, `ING`, `LY`, `S`, `S'`の8件）に一致するかを調べ、一致すればそれを末尾に連結して完了する。
   - 一致しなければ、`remaining`を対象に`_partsToKana`を**再帰呼び出し**する（このときは`includePrefix=False`、つまり2回目以降のマッチでは接頭辞辞書は使わない）。再帰呼び出しが成功すれば、両方の変換結果を連結して返す。失敗すれば、その`cnt`では全体として失敗とみなし、1つ短い`target`で再試行する。

実測で確認した具体例（`converter.process()`の実際の出力とデバッグログより）:

| 入力 | 分解のされ方 | 変換結果 |
| --- | --- | --- |
| `unhappy` | `un`（`PREFIX`）＋`happy`（`PHRASES`） | アンハッピー |
| `tweeting` | `tweet`（`PHRASES`）＋`ing`（`SUFFIX`） | トゥィートゥイング |
| `podcasting` | `pod`（`PHRASES`）＋`casting`（`PHRASES`、再帰呼び出しで解決） | ポッドゥキャスティング |

いずれも、`unhappy`・`tweeting`・`podcasting`という文字列自体は辞書に丸ごと登録されていない（`WORDS`にも`PHRASES`にも完全一致エントリがない）が、部品に分解することで変換できている。

### 4.3 ローマ字変換（`_romanToKana`, L185-261 / `ROMAN`辞書）

`_engToKana`で変換できなかった英字部分（`_partsToKana`が失敗を返し、元の文字列がそのまま残っている箇所）に対して適用される。これは辞書に単語として存在しない固有名詞・造語などを、**ローマ字の読み方に基づいて機械的に**カナへ変換するための最終手段の一歩手前の処理である。

- `ROMAN_MIN=3`文字未満の文字列には適用しない（短すぎる文字列をローマ字として読むと誤読が多くなるための下限）。
- 文字列を先頭から1文字ずつ処理する。処理の各ステップで、まず**促音（っ）判定**を行う。現在位置と次の1文字が同じ子音で、かつその文字が`SOKUON_IGNORE`（`A`, `I`, `U`, `E`, `O`, `N`）に含まれない場合、「ッ」を出力して1文字だけ読み進める（母音やNの連続は促音にしない）。
- 促音でなければ、現在位置の1文字が`ROMAN`辞書に単独で存在するかを調べる。
  - 存在する場合、そこから**後続の文字を取り込みながら**2文字、3文字…と範囲を広げていき、引き続き`ROMAN`にヒットする限り貪欲に長いキーへ伸ばしていく（例:`C`単体ではなく`CHA`のような複数文字の組み合わせを優先する）。
  - 存在しない場合は逆に、残り文字列の**末尾から手前へ**向かって長い部分文字列から順に`ROMAN`辞書と照合し、最初に見つかった（＝可能な範囲で最長の）組み合わせを採用する。
  - どちらの経路でも一致が1つも見つからなければ、**その文字列全体の変換を諦め、元のアルファベットのまま**にする（部分的に変換できていた分も破棄される、いわゆる all-or-nothing）。

実測例:

| 入力 | 結果 | 備考 |
| --- | --- | --- |
| `tanaka` | タナカ | `TA`+`N`+`NA`（一度NA単独に戻る）+`KA`という順で`ROMAN`辞書をマッチ |
| `mokkuna` | モックナ | `K`の連続が促音「ッ」に変換される例 |
| `tekkano` | テッカノ | 同上 |
| `abc` | `abc`（未変換のまま） | `A`は変換できるが続く`AB`/`BC`のいずれも`ROMAN`辞書にヒットせず、全体が失敗してロールバックする例 |

`abc`のように`_romanToKana`でも変換できなかった文字列は、そのままの形で次段階（5章）に渡される。

## 5. 後処理：結合とスペルアウト

### 5.1 チャンクの結合（`_trimWhitespaceBetweenUpperCase`, L263-267）

`_splitUpperCase`で分割したリストを`"".join()`で1つの文字列に戻すだけの処理である。メソッド名は「大文字間の空白を除去する」という意味に読めるが、**現在の実装は単純な文字列結合のみを行っており、空白の除去・挿入は行っていない**（そもそも`_splitUpperCase`は要素間に空白を挿入しないため、除去すべき空白が生じない）。

### 5.2 スペルアウト（`_alphaToSpell`, L269-286 / `SPELL`辞書）

`process()`の最終段階。文字列中に残っているすべての英字（大文字・小文字問わず、1文字単位）を`dictionaries.SPELL`（`A`→エー、`B`→ビー、…`Z`まで26件）で置き換える。`process(s, spellout=False)`とした場合はこの段階がスキップされ、未変換のアルファベットがそのまま出力に残る。

ここまでの4段階（辞書全体一致／複合語分解／ローマ字変換）のいずれでも変換できなかった文字がここで初めて「アルファベットの読み方」として読み上げられる。3.2節で1文字ずつに分解された頭字語（`NHK`→`N`,`H`,`K`など）も、他の手段では変換されようがないため、最終的にここで`エヌ`・`エイチ`・`ケー`という形に変換される。

## 6. 辞書・定数ファイルとアルゴリズムの対応

| ファイル | 参照するメソッド | 役割 |
| --- | --- | --- |
| `dictionaries/words.json` | `_engToKana` | 英字列**全体**が完全一致した場合にのみ使う単語辞書。複合語分解の部品としては使われない |
| `dictionaries/phrases.json` | `_partsToKana` | 複合語分解の基本部品。単独の単語としても、他の部品と組み合わせる形でも使われる（約49,000件と最大） |
| `dictionaries/prefix.json` | `_partsToKana` | 単語の**先頭**にのみ適用される接頭辞（`NO`, `UN`のみ） |
| `dictionaries/suffix.json` | `_partsToKana` | 一致した部品の**直後**にのみ適用される接尾辞（`ING`, `ED`など8件） |
| `dictionaries/roman.json` | `_romanToKana` | 辞書に存在しない語をローマ字読みするための音節対応表 |
| `dictionaries/spell.json` | `_alphaToSpell` | 最終手段のアルファベット読み上げ（A〜Zの26件） |
| `constants/upper_ignore.json` | `_splitUpperCase` | 3文字以下の大文字列のうち、頭字語として分解せず単語として扱いたいものの一覧 |
| `constants/must_spelled.json` | `_partsToKana` | 辞書に一致しても強制的にスペルアウトさせたい文字列の一覧（`PHRASES`より優先） |

## 7. 具体例で見る全体の流れ

`XMLHttpRequest`を例に、全段階を通しで追う。

1. `_zenToHan`: 半角アルファベットのみなので変化なし。
2. `_splitUpperCase`: `['X', 'M', 'L', 'Http', 'Request']`に分割（3.2節参照。`Request`の`R`で新語境界と判定→切り出し、続く`XMLH`のうち`H`が次の`Http`の先頭とみなされ切り出され、残った`XML`は3文字なので1文字ずつ展開される）。
3. 各チャンクを`_engToKana`→`_romanToKana`で変換:
   - `X`, `M`, `L`: 1文字では`WORDS`/`PHRASES`/`ROMAN`のいずれにも一致せず、未変換のまま持ち越し。
   - `Http`: `PHRASES`/`ROMAN`のいずれにも一致せず、未変換のまま持ち越し。
   - `Request`: `PHRASES`に`Request`→`リクウェストゥ`として登録されており、そのまま変換。
4. `_trimWhitespaceBetweenUpperCase`で結合: `XMLHttpリクウェストゥ`。
5. `_alphaToSpell`で残った`X`,`M`,`L`,`H`,`t`,`t`,`p`を1文字ずつスペルアウト: `エックス`+`エム`+`エル`+`エイチ`+`ティー`+`ティー`+`ピー`+`リクウェストゥ`。

最終結果: `エックスエムエルエイチティーティーピーリクウェストゥ`

このように、1つの入力文字列の中でも「辞書一致」「ローマ字変換」「スペルアウト」の3つの変換方式が部分ごとに混在しうる、というのが本アルゴリズムの特徴である。

## 8. 実装上の注意点

* **優先順位は「完全一致 > 部品分解 > ローマ字 > 1文字スペルアウト」の順に固定**であり、より「自然な読み」に近いほど優先される設計になっている。辞書に単語を追加すれば、それだけローマ字変換や機械的なスペルアウトより優先されるようになる。
* `_romanToKana`は**単語単位でall-or-nothing**である。文字列の一部分だけ読めても、途中で読めない箇所にぶつかると、それまでの変換結果を含めて全体が未変換のまま巻き戻る（4.3節の`abc`の例）。部分的にローマ字変換された結果が出力に混ざることはない。
* `MUST_SPELLED`は`_partsToKana`内でのみ判定される。3.2節②のルールにより、3文字以下の大文字列は`UPPER_IGNORE`に載っていない限りそもそも`_splitUpperCase`の時点で1文字ずつに分解されてしまうため、**`MUST_SPELLED`が実際に効果を発揮するのは、小文字表記や、他の単語の一部として埋め込まれた場合など、`_splitUpperCase`で分解されずに`_partsToKana`まで到達したケースに限られる**。
* `PREFIX`辞書はエントリ数が2件（`NO`, `UN`）のみと非常に小さい。README.mdにある通り「単語の先頭にあれば変換するが、途中や末尾では変換しない」という特殊な挙動が必要な場合にのみ使うべき辞書であり、通常の単語登録は`PHRASES`（または`WORDS`）に対して行う。
